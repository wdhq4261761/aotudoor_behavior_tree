# AutoDoor 行为树系统 — 架构文档

## 1 项目概述

### 1.1 项目定位

AutoDoor 行为树系统是一个独立的可视化行为树编辑与执行框架，面向 Windows 平台的自动化场景（游戏辅助、RPA 流程等）。系统提供完整的图形化编辑器、多种检测/动作节点、脚本录制、OCR 识别、DD 虚拟键盘等能力，并支持标准版与 DD 游戏版两种打包模式。

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| 可视化编辑器 | 基于 CustomTkinter 的节点式编辑器，支持拖拽、连线、缩放、框选 |
| 行为树引擎 | 独立线程执行，支持启动/暂停/停止/恢复 |
| 17 种内置节点 | 3 种复合节点 + 5 种条件节点 + 9 种动作节点 |
| 黑板系统 | 观察者模式的数据共享机制，节点间解耦通信 |
| 序列化 | JSON/YAML/TXT 多格式持久化，版本化数据结构 |
| 撤销/重做 | Command 模式，支持 100 步历史 |
| DD 虚拟键盘 | 游戏级硬件模拟输入，绕过大多数输入检测 |
| OCR 集成 | 内嵌 Tesseract OCR，支持中英文识别 |
| 脚本录制 | TXT 脚本录制与回放 |
| 双版打包 | 标准版 (PyAutoGUI) 与游戏版 (DD64.dll) |

### 1.3 技术栈

```
Python 3.10+
├── GUI 框架:    CustomTkinter + Tkinter Canvas
├── 图像处理:    Pillow, OpenCV, imagehash
├── OCR 引擎:    RapidOCR + ONNX Runtime
├── 输入模拟:    PyAutoGUI (标准版) / DD64.dll (游戏版)
├── 音频播放:    pygame.mixer
├── 打包工具:    PyInstaller
└── CI/CD:       GitHub Actions
```

## 3 核心组件设计

### 3.1 节点模型 (bt_core/nodes.py)

节点模型是整个系统的核心抽象，采用经典的 Composite 模式设计：

```
Node (ABC)
├── CompositeNode (ABC)
│   ├── SequenceNode     — 顺序执行，全部成功才成功
│   ├── SelectorNode     — 选择执行，任一成功即成功
│   ├── ParallelNode     — 并行执行，策略决定成功条件
│   └── StartNode        — 根节点，顺序执行子节点，失败后继续执行后续子节点
├── ConditionNode (ABC)
│   ├── _children_running: bool  — 子节点执行状态追踪标志
│   ├── OCRConditionNode     — OCR文字检测
│   ├── ImageConditionNode   — 图像模板匹配
│   ├── ColorConditionNode   — 颜色检测
│   ├── NumberConditionNode  — 数字识别比较
│   └── VariableConditionNode— 变量值判断
└── ActionNode (ABC)
    ├── _children_running: bool  — 子节点执行状态追踪标志
    ├── KeyPressNode     — 键盘按键
    ├── MouseClickNode   — 鼠标点击
    ├── MouseMoveNode    — 鼠标移动（支持平滑移动）
    ├── MouseScrollNode  — 鼠标滚动
    ├── DelayNode        — 延时等待
    ├── SetVariableNode  — 设置变量
    ├── ScriptNode       — 执行脚本
    ├── CodeNode         — 执行代码
    └── AlarmNode        — 播放报警
```

**节点核心接口：**

| 方法 | 说明 |
|------|------|
| `tick(context)` | 执行节点逻辑，返回 NodeStatus |
| `reset(reset_counters)` | 重置节点状态，`reset_counters` 控制是否重置计数器 |
| `abort(context)` | 中止节点执行 |
| `add_child(child)` | 添加子节点 |
| `remove_child(child)` | 移除子节点 |
| `to_dict()` | 序列化为字典 |
| `from_dict(data)` | 从字典反序列化 |

**节点日志输出机制：**

所有节点在执行成功或失败时都会通过 LogManager 输出运行日志：

| 日志类型 | 触发时机 | 输出内容 |
|----------|----------|----------|
| 成功日志 | 节点返回 SUCCESS | 节点类型、节点名称 |
| 失败日志 | 节点返回 FAILURE | 节点类型、节点名称、失败原因 |
| 中止日志 | 节点被 abort() | 节点类型、节点名称 |

**失败原因输出规范：**

| 节点类型 | 失败原因示例 |
|----------|-------------|
| SequenceNode | "子节点 [节点名] 执行失败" |
| SelectorNode | "所有子节点执行失败" |
| ParallelNode | "成功 2/5 个子节点" |
| StartNode | "部分子节点执行失败" |
| OCRConditionNode | "未找到关键词: '登录'，识别到的文本: 注册 取消" |
| ImageConditionNode | "未找到匹配的图像，最高置信度: 45.23%，阈值: 80.00%" |
| ColorConditionNode | "未找到匹配的颜色 #FF0000，容差: 30" |
| NumberConditionNode | "未识别到数字，识别到的文本: ABC DEF" |
| VariableConditionNode | "变量 'count' 不存在" 或 "变量 'count' 值为 5，不满足 > 10" |

**成功日志简化原则：**

成功日志仅输出节点类型和名称，不包含额外信息，保持简洁：

```python
LogManager.instance().log_success(
    node_type="鼠标移动节点",
    node_name="移动到按钮位置"
)
```

**节点状态管理机制：**

节点在重试（retry）和重复执行（repeat）时需要正确管理状态：

| 方法 | 说明 |
|------|------|
| `_reset_for_retry()` | 重试时重置状态，设置 `status = RUNNING`，保留重试计数器 |
| `_reset_for_repeat()` | 重复执行时重置状态，设置 `status = RUNNING`，保留重复计数器 |
| `_execute_with_decorators()` | 装饰器模式处理 retry/repeat/timeout |

**重要：** `_reset_for_retry()` 和 `_reset_for_repeat()` 必须将 `status` 设置为 `RUNNING`，而非 `SUCCESS`。这是因为在重试/重复期间，节点仍处于执行状态，需要返回 `RUNNING` 让引擎继续 tick。

**子节点执行状态追踪机制：**

ActionNode 和 ConditionNode 在执行子节点时需要追踪子节点的执行状态，避免在子节点运行期间重复执行动作/条件检测：

| 属性 | 说明 |
|------|------|
| `_children_running` | 布尔标志，标记子节点是否正在执行 |

**状态追踪流程：**

```
ActionNode/ConditionNode 执行流程:
┌─────────────────────────────────────────────────────────────────────┐
│  1. 首次 tick:                                                       │
│     - 执行动作/条件检测                                               │
│     - 若成功且有子节点:                                               │
│       - 通知 GUI 节点状态为 SUCCESS                                   │
│       - 设置 _children_running = True                                │
│       - 开始执行子节点                                                │
│                                                                      │
│  2. 后续 tick:                                                       │
│     - 检查 _children_running 标志                                    │
│     - 若为 True，跳过动作/条件检测，继续执行子节点                      │
│     - 子节点执行完成后，重置 _children_running = False                │
└─────────────────────────────────────────────────────────────────────┘
```

**GUI 状态通知机制：**

当 ActionNode 或 ConditionNode 成功执行后需要执行子节点时，必须先通知 GUI 节点状态为 SUCCESS，确保界面正确显示节点状态：

```python
# 在执行子节点前通知 SUCCESS 状态
if status == NodeStatus.SUCCESS and self.children:
    context.notify_node_status(self.node_id, "success")
    self._children_running = True
    return self._execute_children(context)
```

**reset 方法签名：**

所有节点的 `reset()` 方法应接受 `reset_counters: bool = True` 参数：

```python
def reset(self, reset_counters: bool = True) -> None:
    """
    重置节点状态
    
    Args:
        reset_counters: 是否重置重试/重复计数器，默认 True
    """
    if reset_counters:
        self._retry_count = 0
        self._repeat_count = 0
    self.status = NodeStatus.SUCCESS
    self._tick_count = 0
    self._start_time = None
```

### 3.2 执行引擎 (bt_core/engine.py)

BehaviorTreeEngine 负责行为树的生命周期管理：

- **执行模式**：独立守护线程循环 tick
- **Tick 间隔**：默认 50ms，可配置
- **状态回调**：`on_status_change` / `on_node_status` 通知外部
- **线程安全**：`threading.Lock` 保护启动/停止操作

**引擎状态机：**

```
    ┌──────────┐
    │  Idle    │ ◄──────────────────────┐
    └────┬─────┘                        │
         │ start()                      │
         ▼                              │
    ┌──────────┐    pause()    ┌────────┤
    │ Running  │──────────────►│ Paused │
    └────┬─────┘◄──────────────└────────┘
         │  resume()                   
         │                             
         ├─ stop() ──────► Idle        
         │                             
         └─ 树执行完毕 ───► Completed ──► Idle
```

### 3.3 黑板系统 (bt_core/blackboard.py)

黑板是节点间通信的核心机制，采用观察者模式：

| 方法 | 说明 |
|------|------|
| `get(key, default)` | 获取变量值 |
| `set(key, value)` | 设置变量值，触发通知 |
| `increment(key, amount)` | 递增数值变量 |
| `delete(key)` | 删除变量 |
| `subscribe(key, callback)` | 订阅变量变化 |
| `unsubscribe(key, callback)` | 取消订阅 |

**内置变量：**

| 变量名 | 说明 |
|--------|------|
| `last_detection_position` | 最近一次检测到的位置坐标（默认，可通过 position_key 自定义） |
| `last_number_value` | 最近一次识别的数字值 |
| `execution_count` | 执行计数 |

**可配置变量名：**

条件节点支持 `position_key` 参数，允许自定义位置保存的黑板变量名，支持多检测点场景。默认值由 `ConfigManager` 管理。

### 3.4 执行上下文 (bt_core/context.py)

ExecutionContext 封装运行时依赖，采用延迟初始化模式：

```
ExecutionContext
├── blackboard: Blackboard          # 黑板系统
├── elapsed_time: float             # 已执行时间
├── tick_count: int                 # tick次数
├── get_screenshot(region)          # 截图（延迟初始化 ScreenshotManager）
├── execute_key_press(key, ...)     # 按键（延迟初始化 InputController）
├── execute_mouse_click(...)        # 鼠标点击
├── execute_mouse_move(position, relative, smooth)  # 鼠标移动（支持平滑移动）
├── get_mouse_position()            # 获取鼠标位置
└── perform_ocr(image, keywords)    # OCR识别（延迟初始化 OCRManager）
```

### 3.5 节点注册中心 (bt_core/registry.py)

NodeRegistry 使用类级别字典管理节点类型映射，支持动态注册：

```python
# 注册节点类型
NodeRegistry.register("SequenceNode", SequenceNode)

# 创建节点实例
node = NodeRegistry.create_node({"type": "SequenceNode", "id": "n1", ...})

# 列出所有类型
types = NodeRegistry.list_types()
```

### 3.6 序列化器 (bt_core/serializer.py)

Serializer 支持三种格式的序列化/反序列化：

| 格式 | 扩展名 | 说明 |
|------|--------|------|
| JSON | `.json` | 默认格式，完整支持 |
| YAML | `.yaml/.yml` | 需要 PyYAML |
| TEXT | `.txt/.bt` | 纯文本树形表示（仅导出） |

**反序列化返回值：**

`Serializer.deserialize()` 返回元组 `(root_node, canvas_state, editor_state)`：
- `root_node`：反序列化的根节点
- `canvas_state`：画布状态（节点位置、视口偏移、缩放比例）
- `editor_state`：编辑器状态（选中节点、文件路径）

引擎加载时仅需 `root_node`，编辑器加载时使用完整元组恢复编辑状态。

### 5.7 UI 更新分发器 (bt_utils/ui_dispatcher.py)

UIUpdateDispatcher 是解决 tkinter 线程安全问题的核心组件，采用事件驱动 + 定时轮询混合模式：

**设计背景：**

tkinter 的 GUI 操作必须在主线程执行，但行为树引擎在独立线程中运行。直接从后台线程调用 `update()` 或 `event_generate()` 可能导致界面假死或更新延迟。

**核心机制：**

| 机制 | 说明 |
|------|------|
| 线程安全队列 | `Queue` 存储待处理的 UI 更新任务 |
| `after(0, callback)` | tkinter 唯一线程安全的调度方法 |
| 定时轮询 | 30ms 间隔检查队列，确保不遗漏更新 |
| 节点状态缓存 | 合并相同节点的重复状态更新 |

**数据流：**

```
后台线程 (Engine)                     主线程 (GUI)
       │                                   │
       │  dispatch_node_status()           │
       │  ┌──────────────────────┐         │
       │  │ Queue.put(task)      │         │
       │  │ after(0, process)    │────────▶│
       │  └──────────────────────┘         │
       │                                   │  _process_updates()
       │                                   │  ┌──────────────────────┐
       │                                   │  │ 从队列取出任务        │
       │                                   │  │ 执行回调更新UI        │
       │                                   │  └──────────────────────┘
       │                                   │
       │  同时：定时轮询检查队列            │
       │  ┌──────────────────────┐         │
       │  │ 每30ms检查队列        │────────▶│
       │  │ 有任务则处理          │         │
       │  └──────────────────────┘         │
```

**关键方法：**

| 方法 | 说明 |
|------|------|
| `attach(widget)` | 绑定到 GUI 组件 |
| `dispatch_node_status()` | 分发节点状态更新 |
| `dispatch_log_flush()` | 分发日志刷新请求 |
| `start_polling()` | 启动定时轮询 |
| `stop_polling()` | 停止定时轮询 |
| `process_pending()` | 立即处理待处理任务 |

- 覆盖：删除现有项目，导入新项目

## 8 设计决策与权衡

| 决策 | 选择 | 原因 |
|------|------|------|
| GUI 框架 | CustomTkinter | 原项目使用，暗色主题原生支持 |
| 画布实现 | Tkinter Canvas | 轻量级，无需额外依赖 |
| 节点通信 | 黑板模式 | 解耦节点间依赖，支持观察者通知 |
| 撤销系统 | Command 模式 | 标准方案，支持细粒度操作回退 |
| 输入抽象 | 延迟初始化 | 避免启动时加载不必要的驱动 |
| 序列化 | 扁平化节点+连线 | 兼容编辑器与引擎两种数据格式 |
| 打包 | PyInstaller | Windows 单文件分发，支持 DLL 打包 |
| 反序列化返回值 | 元组 (node, canvas_state, editor_state) | 支持编辑器完整状态恢复 |
| 快捷键焦点保护 | 检查焦点组件类型 | 避免在输入框中误删节点 |
| 资源路径管理 | ResourceManager 单例 | 统一管理开发/打包环境的资源路径 |
| Tesseract 路径 | 自动检测 | 简化用户配置，支持打包环境 |
| 崩溃恢复 | 静默自动恢复 | 提升用户体验，避免重复确认 |
| 粘贴撤销 | AddNodesCommand | 确保粘贴操作可撤销 |
